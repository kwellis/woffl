    - Pumps selected from E-41 batch semi-finalists.

PRIORITY 1B: Solver Refactor (solopump.py / jetflow.py)
------------------------------------------------------------------
  1. Fix outer secant indexing — jetpump_solver uses psu_list[0] and [1]
     (fixed bracket endpoints) instead of [-2] and [-1]. Single false-
     position step, not a converging secant.

  2. Replace 50-point throat entry sweep with targeted root-find —
     throat_entry_zero_tde builds a full JetBook just to find pte where
     dEte=0. Only the zero crossing is needed during solve. Keep full
     book path for visualization.

  3. Smarter power fluid initial guesses — inner secant starts at
     [2000, 3000] BWPD always. Estimate from nozzle area and pressure
     differential to start closer.

  4. Prune unnecessary stored values — arrays and intermediates from
     learning phase that aren't needed for the final solve.

  5. Consider flattening nested secant loops — outer on psu, inner on
     qpf. Single 2D solve might reduce total evaluations.

PRIORITY 1C: Smart Semi-Finalist Search (batchpump.py)
------------------------------------------------------------------
  Currently batch_run solves every pump in the grid (~60 per well) to
  find ~10 semi-finalists. Most of those 60 runs are wasted — small
  nozzles with small throats fail, big nozzles with big throats fail.

  Idea: walk the grid intelligently instead of brute-forcing it. The
  semi-finalist frontier follows a pattern — small nozzles pair with
  large throat ratios (lots of reservoir pressure helping), and as
  nozzle size increases the throat ratio shrinks (need more of the
  throat for power fluid momentum to lift). Could exploit this:

    - Start at e.g. 9E, solve it
    - Step throat ratio down (9D) — if oil drops, this nozzle's best
      throat is E, move to 10E
    - If oil increases, keep stepping down (9C, 9B...) until oil drops
    - Move to next nozzle size, start from the previous best throat
      ratio and search around it
    - The frontier moves from large-ratio/small-nozzle toward
      small-ratio/large-nozzle, so you're always searching near the
      boundary

  Target: cut from ~60 solves to ~15-20 while still finding all (or
  nearly all) semi-finalists. Even cutting in half would speed up
  batch_run and network optimization significantly.

FUTURE: Import / Package Structure Brainstorm
-----------------------------------------
  Currently using a typical example looks like:

    from woffl.assembly.batchpump import BatchPump
    from woffl.flow.inflow import InFlow
    from woffl.geometry.jetpump import JetPump
    from woffl.geometry.pipe import Pipe, PipeInPipe
    from woffl.geometry.wellprofile import WellProfile
    from woffl.pvt.blackoil import BlackOil
    from woffl.pvt.formgas import FormGas
    from woffl.pvt.formwat import FormWater
    from woffl.pvt.resmix import ResMix

  That's 9 import lines before you write a single line of code. Pandas
  gets away with "import pandas as pd" and you have the whole library.
  Numpy same thing. We have flat modules (one class per file) which is
  clean to navigate but verbose to import.

  Questions to work through:
    - Can __init__.py re-exports get us to something like:
        from woffl.pvt import BlackOil, FormGas, FormWater, ResMix
        from woffl.geometry import JetPump, Pipe, PipeInPipe, WellProfile
      (some of these already work — need to audit and make consistent)
    - Could we go even flatter? "from woffl import *" or "import woffl"
      with woffl.JetPump, woffl.BatchPump etc? Tradeoff: convenient but
      hides where things live, harder to navigate the source.
    - What's the right balance between "easy to use" and "easy to find
      where code lives"? The subpackages (pvt, geometry, flow, assembly)
      are good logical groupings — maybe the answer is just better
      __init__.py files, not restructuring.
    - Look at how other small scientific Python packages handle this.

FUTURE: SoloPump Class (analysis vs speed split)
-----------------------------------------
  Right now jetpump_solver returns a lean tuple (psu, sonic, qoil, fwat,
  lwat, mach) and throws away all the intermediate physics. That's great
  for BatchPump speed, but bad for understanding a single well.

  Idea: add a SoloPump class in solopump.py that captures the full
  picture for a single well + single pump:

    solopump.py:
        jetpump_solver()    — fast, returns tuple (used by BatchPump)
        class SoloPump      — rich, stores intermediates (used for analysis)

  What SoloPump would store:
    - Throat entry book (dEte curves, mach, velocities)
    - Diffuser book (pressure recovery)
    - Power fluid iteration history
    - Nozzle inlet pressure, throat pressures, momentum balance
    - Outflow pressure profile along the tubing

  Methods:
    - .solve()              — calls jetpump_solver, then builds books
    - .plot_throat_entry()  — 4-panel throat entry visualization
    - .plot_diffuser()      — 4-panel diffuser visualization
    - .plot_pressure_profile() — pressure vs depth along wellbore
    - .summary()            — prints discharge_check style text output

  This would absorb what choked_figures(), discharge_check(), and
  pump_pressure_relation() in jetgraphs.py currently do — but as
  methods on a class that holds state, instead of standalone functions
  with 10+ arguments each. Could eventually make jetgraphs.py obsolete.

  Key principle: jetpump_solver stays fast and stripped down for batch
  optimization. SoloPump wraps it for when you want to understand what's
  happening inside one well.

FUTURE: Fast Solver Path (strip JetBooks from solve loop)
-----------------------------------------
  Per jetpump_solver call, the code builds ~8-15 JetBooks and ~3-5
  outflow profiles, then throws away all but a few scalars:

  Call chain overhead per discharge_residual call:
    1. throat_entry_zero_tde — steps pressure down 25 psi at a time,
       builds JetBook (np.append on 8 arrays per step, ~10-30 steps),
       only uses 4 scalars from dete_zero()
    2. diffuser_discharge — same pattern, steps pressure up 100 psi,
       builds another JetBook, only uses interpolated pdi scalar
    3. production_top_down_press — walks entire wellbore building
       md_seg, prs_ray, slh_ray arrays, only uses prs_ray[-1]
    4. throat_discharge — secant on momentum balance, stores lists
       (relatively lean already)

  And psu_minimize calls throat_entry_mach_one 3-5 times on top of
  that, each building yet another JetBook.

  Fast path would replace these with scalar-only versions:
    - throat_entry_zero: step with scalar tracking (prs, vel, rho,
      snd, tde), stop at zero crossing, no JetBook, no np.append
    - diffuser_discharge_fast: same — step until tde crosses zero,
      interpolate, return scalar
    - outflow_bottom_pressure: walk wellbore, only return final
      pressure, no arrays

  Important: keep the array-building versions intact for SoloPump
  visualization. The fast versions are parallel implementations, not
  replacements. Consider a clean pattern for this — maybe a "store"
  flag, or separate function names (_fast suffix), or SoloPump calls
  the rich versions while jetpump_solver calls the lean ones.

  Pre-profile estimate was 30-50% speedup from JetBook cleanup. Actual
  profiling told a different story — see below.

PROFILING RESULTS (cProfile, 10x jetpump_solver, 12C pump, E-41 well)
----------------------------------------------------------------------
  Total: 1,840,226 function calls in 0.779 seconds (78ms per solve)

  Breakdown by cumulative time:

    56% — production_top_down_press (outflow.py)
           2,640 calls to beggs_diff_press inside. Called 4x per solve
           (once per discharge_residual). Builds md_seg, prs_ray,
           slh_ray arrays walking the entire wellbore, but only
           prs_ray[-1] is ever used. This is the dominant cost by far.

    29% — psu_minimize → throat_entry_mach_one (jetflow.py)
           50 JetBook builds, 2,200 JetBook.append calls (each does
           np.append on 8 arrays = full copy each time). Called at the
           start of every solve to find minimum suction pressure.

     9% — throat_entry_zero_tde (jetflow.py)
           40 more JetBook builds, called inside discharge_residual.
           Only 4 scalars from dete_zero() are actually used.

     4% — diffuser_discharge (jetflow.py)
           Cheapest of the bunch. Builds a small JetBook, ~5-10 steps.

     -- — gas_solubility (blackoil.py)
           Called 102,030 times across everything. Deepest hot function.
           Hit from every PVT condition() call in every loop. Not a
           single bottleneck but adds up everywhere.

  Key takeaways:

    1. OUTFLOW IS THE #1 TARGET, NOT JETBOOKS.
       Initial assumption was that JetBook array overhead was the main
       cost. Wrong — Beggs & Brill pressure traversal is 56% of solve
       time. A fast-path outflow_bottom_pressure() that skips array
       storage and only returns the final pressure would be the single
       biggest win.

    2. Reducing wellbore segments would compound the gain.
       2,640 beggs_diff_press calls for 40 outflow traversals = 66
       segments per walk. If we could get away with fewer segments
       (coarser grid on straight sections, finer near the pump), that
       directly reduces both outflow and PVT call counts.

    3. JetBook cleanup is still worth doing, but it's #2 priority.
       psu_minimize + throat_entry_zero_tde together are ~38%. The
       np.append pattern (copy 8 arrays every step) is wasteful, but
       less wasteful than walking Beggs & Brill 4 times per solve.

    4. PVT caching could cut across everything.
       gas_solubility alone is called 102k times. If condition() cached
       results for repeated P/T values (same pressure step hit from
       multiple callers), it would reduce work in both the outflow and
       JetBook loops. Deeper change, but high leverage.

    5. Focus energy in this order:
         a. Fast outflow (56% of time)
         b. Lean throat entry / psu_minimize (29% + 9%)
         c. PVT caching (cross-cutting, harder to implement)
         d. Diffuser (only 4%, low priority)
